Which implementation approach?

There's already a WPF/ folder with PowerShell+XAML files started. Which direction do you want to go?

A

Continue & complete the PowerShell + XAML approach

Build on the existing WPF/*.ps1 files. Runs by double-clicking a .ps1 file — no compilation, no install. Uses PowerShell's built-in WPF hosting to load XAML windows. The existing files already have the dark theme and main window layout.

Pros

  • Already partially built
  • Zero install — ships as scripts
  • Easy to edit & iterate

Cons

  • PowerShell WPF has limits (no MVVM, no data binding)
  • Can feel clunky for complex interactions
B

Build a proper C# WPF application

A real .NET WPF project (.csproj) that calls the Azure CLI via Process. Produces an .exe. Full MVVM, data binding, async operations, and a polished native Windows app feel. The Python logic gets re-implemented in C#.

Pros

  • Full WPF power (MVVM, async, data binding)
  • Ships as a standalone .exe
  • Best long-term maintainability

Cons

  • More initial build work
  • Requires .NET SDK to build
C

Python GUI (keep Python, add a window)

Add a GUI layer on top of the existing Python codebase using tkinter (built-in) or PySimpleGUI / CustomTkinter. Keeps all the existing Python logic. Runs via python deploy_gui.py.

Pros

  • Reuses all existing Python code
  • No rewriting logic

Cons

  • Python GUIs can look dated
  • Still requires Python installed
  • Not a "true" WPF
